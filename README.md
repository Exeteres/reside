## Реплики

Начнем по порядку. Реплики - это автономные агенты, (микро)сервисы, деплойменты, приложения, you name it,
которые выполняют определенные функции от инфраструктурных (управление другими репликами, мониторинг) до прикладных (как правило, развлекательных).
Концептуально каждая реплика имеет состояние, выраженное, как минимум, в факте ее существования,
хотя ничто не мешает ей уметь горизонтально масштабироваться, если она захочет.

Реплики намеренно одушевлены просто прикола ради.
Если в реальности какой-то сервис называется `telegram-proxy`, то в ReSide это будет Telegram Replica или Телеграмная Реплика. А еще у нее будет анимешный аватар женского пола с характерной символикой. Почему женского, я надеюсь, объяснять не нужно.

## Kubernetes

Реплики живут в Kubernetes-кластере. Каждая реплика в своем собственном неймспейсе. Совокупность неймспейса `reside-system` и неймспейсов реплик называется ReSide-кластером.

В теории, каждая реплика может быть развернута как угодно, но обычно она представляет собой [Knative-сервис](https://knative.dev/docs/serving/convert-deployment-to-knative-service/#knative-service), состояющий из одного контейнера. Почему не обычный деплоймент? Очень просто - реплик предполагается много и каждая из них жрет память. Knative же позволяет репликам спать (scale to zero), когда они не нужны, и просыпаться по запросу.

Каждая реплика также имеет кубовый SA с админскими правами на весь ее неймспейс, который привязан к ее подам. Это позволяет ей творить в своем неймспейсе все, что она захочет. Например, она может создавать конфиги и секреты для хранения всяких токенов и настроек. Или создать CronJob для своих задач. Или может развернуть рядом с собой Redis, если очень надо. Или целую базу данных. А может под с Kubernetes-кластером внутри, в котором развернет еще один кластер ReSide. Это, кстати, не шутка, такое может быть полезно для интеграционных тестов.

## Reside Operator

Как реплика появляется в кластере? Что ж, здесь градус неадеквата немного повышается. Для создания реплик существует `reside-operator` и cluster-wide CRD `Replica`. Такие CRD выглядят примерно так:

```yaml
apiVersion: reside.io/v1
kind: Replica
metadata:
  name: telegram-replica
spec:
  image: ghcr.io/exeteres/reside/replicas/telegram:latest
```

> Для справки: `reside-operator` не одушевлен. Не пытайтесь заговорить с ним. Он намеренно стоит в стороне от всего этого театрализованного безумия и занимается только тем, что смотрит за CRDшками и создает стартовые ресурсы для новых реплик. Ему вообще [нет дела](https://ru.wikipedia.org/wiki/%D0%94%D0%B5%D0%B8%D0%B7%D0%BC) до того, что там внутри происходит.

Что входит в стартовый набор реплики?

- неймспейс с именем `replica-<replica-name>`;
- SA с админскими правами на неймспейс и RoleBinding для него;
- Job с единственным контейнером, в котором указан `image` из CRD.

Всё. Больше оператор ничего не делает. Созданная джоба самостоятельно создает все, что ей нужно.

## Аутентификация

Помимо доступа к Kubernetes API, SA реплик служит для еще одной цели - аутентификации в API других реплик. Реплика использует кубовый TokenRequest API, чтобы получить OIDC ID-токен для своего SA и затем ходит с ним в API других реплик. Эти реплики, в свою очередь, запрашивают публичный ключ из кубового JWKS API, чтобы проверить этот токен.

## Деплой реплики rate

Для целевого выката реплики `rate` используйте штатный bootstrap flow из `apps/cli`.
Ниже минимальный runbook без ручного применения CRD/манифестов:

```bash
bun apps/cli/src/main.ts bootstrap --context <kube-context> --replica rate
```

Реплика `rate` зависит от `infra`, `access` и `telegram`, поэтому без флага `--only`
они будут выбраны и развернуты автоматически.

После запуска проверьте статус:

```bash
kubectl get replica rate -o yaml
kubectl -n replica-rate get pods
```

Если нужен откат к удалению реплики из кластера, удалите её CR:

```bash
kubectl delete replica rate
```

Для повторного выката после отката снова выполните bootstrap-команду выше.
